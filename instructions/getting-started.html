Beginner Guide

Clone this project & re-publish it under your own GitHub account

By the end of this guide your Mac will have a working copy of trackAndFieldWeeklyPlanner, every link to the original repo will be wiped, and a brand-new repo under your own GitHub account will hold an exact copy of the code.

What you'll do
  1. Make sure your Mac has the tools
  2. Step 1 — Clone the repo
  3. Step 2 — Wipe the link to the original repo
  4. Step 3 — Create a new repo under your GitHub account
  5. Step 4 — Push the project to your new repo
  6. Step 5 — Install Node.js and run the app on your Mac
  7. Doing all of this from inside Codex / Cursor
  8. Troubleshooting

0 Make sure your Mac has the tools

You need three things: the Terminal app, git, and (recommended) the GitHub CLI gh. Check what's already installed before installing anything.

Open the Terminal

Press ⌘+Space, type Terminal, hit Return. Everything below is typed into that window.

Check git

git --version

If you see something like git version 2.39.x, you're good. If macOS prompts you to install the Xcode Command Line Tools, click Install and wait a few minutes.

Check the GitHub CLI (optional but easiest)

gh --version

If gh is missing, install it with Homebrew:

# If you don't have Homebrew yet, install it first from https://brew.sh brew install gh

Sign in to GitHub from the CLI

gh auth login

Follow the prompts: choose GitHub.com → HTTPS → Yes, authenticate Git with my GitHub credentials → Login with a web browser. A one-time code will appear; paste it into the page that opens. When it finishes, run:

gh auth status

You should see ✓ Logged in to github.com account <your-username>.

Tip You can do everything in this guide without gh by creating the new repo on github.com in your browser. The gh CLI just makes it faster.

1 Clone the repo to your Mac

"Cloning" downloads a copy of the project into a new folder on your Mac.

Pick where to put it

Most people use a folder like ~/workspace or ~/Projects. Move there first:

# Create the folder if it doesn't exist, then go into it mkdir -p ~/workspace cd ~/workspace

Clone

git clone https://github.com/jerrywang951027/trackAndFieldWeeklyPlanner.git

You'll see something like:

Cloning into 'trackAndFieldWeeklyPlanner'... remote: Enumerating objects: 62, done. Receiving objects: 100% (62/62), done.

Move into the project folder

cd trackAndFieldWeeklyPlanner

Verify

ls git remote -v

You should see all the project files, and git remote -v should show origin pointing at the original repo URL. That's exactly what we'll erase next.

2 Wipe every trace of the original git repo

The folder you just cloned has a hidden .git directory inside it that holds every commit, every branch, and the link back to the original GitHub repo. Deleting it gives you a clean slate.

Make sure you are in the project folder Before running the next command, double-check that you are inside trackAndFieldWeeklyPlanner, not somewhere else. Run pwd first.
pwd

The output should end with /trackAndFieldWeeklyPlanner. Now wipe the git data:

rm -rf .git

Verify

git status

You should now see:

fatal: not a git repository (or any of the parent directories): .git

That error is the goal — it means this is just a folder of files now, with no connection to the original repo, no commit history, no remotes. Perfect.

What this actually removed Every commit message, the entire history, the branches, the tags, and the origin URL pointing at the original repo. The source code itself is untouched.

3 Create a new repo under your own GitHub account

Pick one of the two options below. Option A (CLI) is faster; Option B (browser) is more visual.

Option A — Use gh in the Terminal (recommended)

Stay inside the project folder, then run (replace my-cool-repo with whatever name you want):

gh repo create my-cool-repo --public --description "My copy of the sprint training planner"

For a private repo, swap --public for --private. You'll see something like:

✓ Created repository <your-username>/my-cool-repo on GitHub https://github.com/<your-username>/my-cool-repo

Copy that URL — you'll need it in step 4.

Option B — Use the GitHub website

  1. Open https://github.com/new in your browser.
  2. Fill in the Repository name (e.g., my-cool-repo).
  3. Choose Public or Private.
  4. Important: leave every checkbox off — do not add a README, .gitignore, or license. The new repo must start completely empty.
  5. Click Create repository.
  6. On the next page, copy the HTTPS URL (it looks like https://github.com/<your-username>/my-cool-repo.git).
Why "completely empty" If GitHub auto-creates a README in the new repo, your first push will be rejected because the local history doesn't match. Empty = easy.

4 Push the project to your new repo

Five commands and you're done. Make sure you're still inside the trackAndFieldWeeklyPlanner folder.

1) Start a fresh git repo

git init -b main

This creates a new .git directory with main as the default branch.

2) Stage every file

git add .

Don't worry — secrets and build artifacts are already excluded by the project's .gitignore. You can confirm with git status and look for any file ending in .env (there shouldn't be any).

3) Make the first commit

git commit -m "Initial commit"

4) Connect the local repo to your new GitHub repo

Replace YOUR-USERNAME and my-cool-repo with your actual values:

git remote add origin https://github.com/YOUR-USERNAME/my-cool-repo.git

5) Push

git push -u origin main

You should see:

To https://github.com/YOUR-USERNAME/my-cool-repo.git * [new branch] main -> main branch 'main' set up to track 'origin/main'.

Verify it landed

Open your new repo's URL in a browser. You should see the project files. If you ever want to confirm from the Terminal:

git remote -v git log --oneline
From now on, the normal git workflow applies Edit files → git add <files> → git commit -m "what I changed" → git push. No more -u origin main needed; the upstream is already set.

5 Install Node.js and run the app on your Mac

The project is a Next.js web app. To run it on your Mac you need Node.js (which comes bundled with npm, the package installer). After that it's three commands: install dependencies, add an API key, start the dev server.

5.1 — Install Node.js

You have two ways to install Node. Option A (nvm) is what most developers use because it lets you switch Node versions per project. Option B is a one-click installer if you'd rather not deal with the Terminal.

Option A — Install via nvm (recommended)

Run this in the Terminal to download and install nvm (Node Version Manager):

curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.40.1/install.sh | bash

Then close and reopen the Terminal window (so it picks up the new nvm command). Verify it's there:

command -v nvm

You should see nvm printed. Now install the latest LTS (long-term support) Node and tell your shell to use it:

nvm install --lts nvm use --lts nvm alias default lts/*

The last line makes that LTS version the default in every new Terminal window.

Option B — Use the official installer

  1. Open https://nodejs.org/en/download.
  2. Click the macOS LTS installer (.pkg file).
  3. Open the downloaded file, click through the wizard, finish.

5.2 — Verify Node and npm

node --version npm --version

You should see something like:

v22.11.0 10.9.0

This project requires Node 20 or newer. Anything v20.x or higher (including v22.x) is fine.

5.3 — Move into the project folder

If you're not already in it (you might have closed Terminal since the earlier steps):

cd ~/workspace/trackAndFieldWeeklyPlanner

5.4 — Install the project's dependencies

npm install

This reads package.json, downloads every library the app uses (React, Next.js, Tailwind, the Anthropic SDK, etc.) into a folder called node_modules/, and writes a lock file. Be patient — first time can take 1–3 minutes. Output looks like:

added 421 packages in 47s 129 packages are looking for funding

The "funding" line is informational — ignore it. As long as you see added <number> packages and no red error text, you're good.

What is node_modules? The folder of downloaded libraries. It's huge (hundreds of megabytes), which is why the project's .gitignore excludes it. You never check it in — anyone who clones the repo just runs npm install to recreate it.

5.5 — Set up your environment file

The app talks to Anthropic's Claude model and needs an API key. Copy the example file and edit it:

cp .env.example .env.local

Open .env.local in any editor. From the Terminal you can use TextEdit:

open -e .env.local

You'll see three lines. Replace the placeholder value with your real Anthropic API key:

ANTHROPIC_API_KEY=sk-ant-...your-key-here... ANTHROPIC_BASE_URL=https://eng-ai-model-gateway.sfproxy.devx-preprod.aws-esvc1-useast2.aws.sfdc.cl ANTHROPIC_MODEL=claude-opus-4-7

Where do I get an API key?

  • If you work at Salesforce and use the internal model gateway, your team's preprod gateway already accepts your existing key — leave ANTHROPIC_BASE_URL as-is.
  • If you want to use Anthropic's public API, sign up at console.anthropic.com, create a key, and delete the ANTHROPIC_BASE_URL line entirely (or set it to https://api.anthropic.com).

Save the file (⌘+S) and close the editor.

Never commit .env.local The project's .gitignore already excludes it, so git add . won't pick it up. But if you ever paste an API key into a different file by accident and push it, rotate the key immediately.

5.6 — Start the development server

npm run dev

Output looks like:

▲ Next.js 16.x - Local: http://localhost:32026 ✓ Ready in 2.3s

5.7 — Open the app in your browser

Open http://localhost:32026. On the very first visit you'll be redirected to /onboarding — fill in the 6-step wizard with your athlete profile, click finish, and you'll land on the dashboard. From there click Generate this week's plan on the home page.

The first plan generation is slow on purpose The default model is claude-opus-4-7, which can take 30–120 seconds to produce a structured 7-day plan. The button shows a live elapsed timer so you know it's still working. To make it faster, edit .env.local and switch ANTHROPIC_MODEL=claude-sonnet-4-5, then restart with npm run dev.

5.8 — Stop the dev server

In the Terminal window where it's running, press Ctrl+C. The server stops; you can close the window.

5.9 — Reset the local data (optional)

The app stores everything (your profile, plans, logs, readiness check-ins) in a single JSON file at .data/store.json. To wipe it and start fresh:

npm run db:reset

Next time you start the app, you'll be sent back to onboarding.

Daily routine after the first run

Once the app is set up, working on it is just two commands:

cd ~/workspace/trackAndFieldWeeklyPlanner npm run dev

Open http://localhost:32026, do your thing, Ctrl+C when done.

★ Doing all of this from inside Codex / Cursor

If you're using an AI coding agent (Cursor, Codex CLI, etc.) you don't have to type the commands yourself — describe the goal and let the agent run them. Here's a prompt that works.

Prerequisite

Open the agent's terminal once and run gh auth login so it can push on your behalf without asking for a password every time. After that, a single instruction is enough.

Copy-paste this prompt into Codex / Cursor

# Replace the placeholders before sending I have an existing folder with a project I cloned from someone else's GitHub repo. Please do the following inside that folder: 1. Verify I am inside the right folder by running `pwd` and showing me the result. 2. Delete the existing `.git` directory so the project is no longer tied to the original repo. 3. Create a new GitHub repo under my account named my-cool-repo using the `gh` CLI. Make it public (or `private`). 4. Run `git init -b main`, stage all files, commit with the message "Initial commit", add the new repo as `origin`, and push to `main`. 5. After the push, confirm by showing me the output of `git remote -v` and `git log --oneline -1`. 6. Then help me run the app locally: confirm `node --version` is 20+ (install Node via nvm if missing), run `npm install`, copy `.env.example` to `.env.local` and pause so I can paste my Anthropic API key, then run `npm run dev` and tell me which URL to open. Before pushing, double-check `git status --short` to make sure no `.env*` files or other secrets are about to be committed. If anything looks suspicious, stop and ask me.
Why the "stop and ask me" line matters AI agents will happily push whatever you tell them to. The safety check is your insurance policy in case a stray .env file made it into the folder.

What the agent will do

  1. Run pwd and show you the path.
  2. Run rm -rf .git (after asking for permission, since it's destructive).
  3. Run gh repo create <name> --public.
  4. Run the five git commands from Step 4 above.
  5. Print the verification output for you to confirm.

This is exactly what was done to publish the original repo, so the steps are battle-tested.

? Troubleshooting

git: command not found

macOS hasn't installed the Xcode Command Line Tools yet. Run xcode-select --install and follow the prompt.

gh: command not found

Either install with brew install gh, or skip the CLI and use the GitHub website (Step 3, Option B).

Push is rejected with "non-fast-forward"

The new GitHub repo wasn't empty — it probably has a README. Easiest fix: delete the new repo on GitHub, recreate it with no initial files, and re-run git push -u origin main.

Push asks for a username and password

GitHub no longer accepts passwords over HTTPS. Run gh auth login, choose HTTPS and authenticate Git with my GitHub credentials when asked. After that, pushes work silently.

I see fatal: not a git repository in step 4

You skipped git init -b main, or you ran it from the wrong folder. Run pwd to check where you are, then cd back into the project folder and start step 4 from the top.

I accidentally pushed a secret

Don't panic, but treat the secret as compromised. Rotate it (e.g., regenerate the API key), then either delete the GitHub repo and start over, or use git-filter-repo to remove the file from history and force-push.

node: command not found or npm: command not found

Node didn't install correctly, or your Terminal hasn't picked up the new PATH. Close the Terminal completely (⌘+Q) and reopen it, then re-run node --version. If still missing and you used nvm, run source ~/.nvm/nvm.sh. If you used the official installer, restart your Mac.

npm install fails with EACCES or permission errors

Don't use sudo npm install — that's how npm gets broken. Instead, switch to nvm (Step 5.1, Option A); it installs Node into your home directory where no permissions are needed.

Port 32026 is already in use

Another copy of the dev server is still running somewhere. Find and kill it:

lsof -ti :32026 | xargs kill -9

Then re-run npm run dev.

The terminal warns: ANTHROPIC_API_KEY is not set

Either you didn't create .env.local, you saved it as .env.local.txt (TextEdit sometimes adds the extension), or the key value is empty. From the Terminal:

ls -la .env.local cat .env.local

The first command should show the file exists and is named exactly .env.local. The second should show your key (not the placeholder text). Restart the dev server (Ctrl+C, then npm run dev) — env files are read once on startup.

"Generate plan" spins forever and nothing happens

It's not actually stuck — Claude Opus can take up to two minutes for a structured 7-day plan. Check the Terminal where npm run dev is running; if you see request activity, just wait. To make it noticeably faster, edit .env.local and set ANTHROPIC_MODEL=claude-sonnet-4-5, then restart with npm run dev.

Browser shows "This site can't be reached"

Make sure the dev server is still running. The Terminal window where you ran npm run dev should be showing logs, not a blank prompt. If it crashed, scroll up in that window — the error message will be near the bottom.
